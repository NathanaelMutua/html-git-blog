2. What is Git

Git is the most popular version control system, it really helps us keep track of different versions of a given project or document.

Introduction to Version Control

Version control, as it says itself is all about keeping track of versions - That's prolly how I'd put it in laymans terms.

Version control systems, which I found out are sometimes called source code management(SCM) or revision control systems(RCS)

Version control allows many members to work collaboratively on a project, even of they are not in the same place.

Versioning

As I've come to learn versioning is the creation and management of multiple product releases

This is signified through versioning numbers, such as: v2.2.1

Meaning of each digit
  • "v2..2.1": This is a MAJOR change in a version
  • "v2.2.1": This usually represents a patch release, usually having minor features.
  • "v2.2.1": This represents a patch release, it is usually just basic bug fixes

History

Git was created by Linus Torvald, the creator of the Linux Kernel, in 2005 in response to a liscence dispute with BitKeeper - a version control system at the time. Linus Torvald created Git in 10 days to provide a free, fast alternative to BitKeeper

On December 21st, 2005, Git had its first major release.

Today(on the day of publishing), the latest version of Git is 2.49.0

Why Use Git

  1. Helps us keep track of changes made by different members
  2. Tracking of changes makes going back to a stable version easier.
  3. It ensures better communication and organization
  4. Allows for many members to work on the same project - collaboration.
  5. Branching and merging allows people to create seperate branches, which they can work on seperately, before requesting to be merged to the main branch - combine changes.

4. Configuring Git

I wil briefely talk about basic configurations, but for detailed information you can visit git

A list of all the available cofiguration variables can easily be accessed using the git help --config

Configuration Levels

In my google research I discovered that you can configure settings to apply to a specific area selectively, as such listed:

  • System-Wide Configuration
  • User-Specific Configuration
  • Repository -Specific Configuration

System-Wide Configuration

These settings, as I have discovered apply to every user on the system

Usually declared as follows: git config --system 'key' 'value'

User-Specific Configuration

These settings apply to a single user and are stored in the /.gitconfig file.

It uses the global flag: --global

Declared as: git config --global 'key' 'value'

Repository -Specific Configuration

These settings apply only to the current repository and are stored in the .git/config file

Declared as: git config 'key' 'value'

Basic Configuration Commands

We will be going through the basics, namely:

  • Setting username and email
  • Customizing default text editor

Setting username and email

One usually needs to identify themselves before making any commit, this is usually the author's name and email address

To declare that identity for all repositories we use git config --global

This will store the settings in the user's .gitconfig file

We will use the following to configure username and email:

  • git config --global user.name "Your Name"
  • gi config --global user.email examplemail@mail.com

Customizing Default text editor

Customizing and configuring the prefered text editor, let's us determine what we will use for various tasks such as writting and editing commit messages

Here we will use the command below:

  • git config --global core.editor "Name of Editor"

For configuring the default editor for a specific repository we use the following command. As seen previously, this usually doesn't have the global flag

  • git config core.editor "Name of Editor

The names of editors can be as follows:

  • VS Code: "code --wait"
  • vim: "vim"
  • emac: "emac"...I guess this is a type of editor - Though I have not really heard about it

5. Initializing a Git Repository

A repository, as we were taught, is a workplace which tracks and manages files within a folder. We can have as many repos as we want on our machine, all with seperate history and content, however, it is important to note that the project should have a single Git repository

Using git init

To initialize a git repository, we run the following command, in the command prompt with the target file directory open

Command: git init

This command is only done once, this creates the .git file in the target repo

Understanding .git directory

To confirm whether the project has a git repository or not, we use the git status command. when you run the git status command and get the message "fata: not a git repository...", it means the project lacks a git repo. You can go ahead and initialize a repo.

6. Adding files to the staging area

This generally means preping modified files for a commit. This allows us to organize our changes into meaningful groups before commiting them

Before we add anything to the staging area, our instructor informed us that we must first confirm if there is anything to add using git status

Using git add . command

This command stages all the changes.

Command: git add .

Adding specific files

To stage specific files, we use the format 'git add "file_path"'

Command: git add file.txt

Adding all changes

To stage all files we use git add -a, which stages all changes, modifications and new files

  • Command: git add -a

On the other hand, I came to realise that if you only want to stage modified and deleted files only, we can use the -u flag

  • Command: git add -u

7. Commiting Changes

This is a really important part, a commit is generally a checkpoint to your project at a particular time

we use the commit function

Writing Commit Messages

This involves having a message to accompany the checkpoint, it let;s others know what changes were made and what to expect

There are 2 ways to make commit messages

  • Direct commit message: this is usually a short message entered in the terminal
  • Detailed text editor commit message: this usually needs too be edited in the selected text editor

Here are the commands to execute the above respectively:

  • Direct commit message: git commmit -m "Message to be filled here"
  • git commit

Amending commits

I did some research, and found that this is a way to modify the most recent commit made in the branch

We can do so like this: git commit --amend -m "New updated commit message"

8. Branches

Branches are like alternative timelines of a project

9. Creating a Branch

We can create a branch using the command git branch NEW_BRANCH_NAME

10. viewing all branches

To view all branches we carry out the command git branch, this gives a list of all branches, with the active branch being highlighted.

11. Switching active branches

To switch between branches we use the command git switch TARGET_BRANCH_NAME

12. Merging a branch

to merge a branch we first relocate/switch to the branch to which we want the target branch to be merged to, then we run the following command:

  • git merge TARGET_BRANCH_NAME

13. Deleting a branch

To delete a branch we run the command git -d branch
